# CHANGELOG.md Setup

Create and maintain CHANGELOG.md following Keep a Changelog format.

## Location
`CHANGELOG.md` in project root

## Format
Follow [Keep a Changelog](https://keepachangelog.com/):

```markdown
# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- (new features go here during development)

### Changed
- (changes to existing functionality)

### Deprecated
- (features that will be removed)

### Removed
- (features that were removed)

### Fixed
- (bug fixes)

### Security
- (security-related changes)

## [0.1.0] - 2024-XX-XX

### Added
- Initial release
- Authentication commands (login, status, logout, doctor, ping, setup)
- Account commands (list, refresh)
- Transaction commands (list, update)
- Budget commands (list)
- Cashflow commands (summary)
- Category commands (list)
- Multiple output formats (json, table, csv, compact, ndjson)
- Date presets for transaction filtering
- Dual-backend session storage (keyring + file)
- Shell completions for Bash, Zsh, Fish
- AI agent optimizations (quiet mode, batch operations)

[Unreleased]: https://github.com/yourname/monarch-cli/compare/v0.1.0...HEAD
[0.1.0]: https://github.com/yourname/monarch-cli/releases/tag/v0.1.0
```

## Guidelines

### When to Update
- Add entries as changes are made (not all at once before release)
- Keep "Unreleased" section current during development
- Move entries to versioned section at release time

### Entry Format
- Use present tense ("Add feature" not "Added feature")
- Include issue/PR numbers when relevant: "Fix crash on empty response (#42)"
- Be specific enough to be useful

### Breaking Changes
Clearly mark breaking changes:
```markdown
### Changed
- **BREAKING**: Renamed `--verbose` flag to `-v` (short form only)
```

### Links
- Link version numbers to GitHub releases/tags
- Link Unreleased to comparison with latest release

## Acceptance Criteria

- [ ] CHANGELOG.md created in project root
- [ ] Follows Keep a Changelog format
- [ ] Has Unreleased section
- [ ] Has initial 0.1.0 version documented
- [ ] All MVP features listed
- [ ] Links to GitHub releases
